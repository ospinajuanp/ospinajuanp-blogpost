# 🚀 ¿Cómo asegurar que el proyecto no falle?

Validar requerimientos no es más que sentarse a confirmar que lo que entendimos y escribimos es exactamente lo que el cliente tiene en la cabeza. Si nos saltamos este paso, cualquier error se arrastra como una bola de nieve y arreglarlo después sale carísimo.

---

# 🔍 El Checklist de un Requisito de Valor

Para saber si un requisito está bien hecho, pásale este filtro rápido:

* **¿Es Válido?**: ¿Realmente soluciona un problema del cliente?
* **¿Es Consistente?**: ¿No se contradice con lo que pedimos en la página anterior?
* **¿Está Completo?**: ¿Incluye todas las reglas y restricciones del usuario?
* **¿Es Realista?**: ¿Tenemos el tiempo, la plata y la tecnología para hacerlo?
* **¿Se puede probar?**: Si no hay forma de medir si funciona, no nos sirve.

---

# 🛠️ Técnicas para no fallar en el intento

### 1. Revisiones (El "Ojo" Humano)
Reunirse con el equipo y el cliente para leer los documentos. Puede ser desde una charla con café para ver si "suena bien", hasta una inspección formal con lupa para detectar fallos lógicos.

### 2. Prototipos (Ver para Creer)
Hacer dibujos o maquetas interactivas antes de programar una sola línea de código. 
* **Lo bueno:** El cliente entiende mejor un dibujo que 50 páginas de texto.
* **Ojo aquí:** A veces el cliente ve el dibujo y cree que el sistema ya está casi listo.

### 3. Casos de Prueba
Si desde el día uno pensamos en: *"¿Cómo voy a probar que esto funciona?"*, el requisito queda mucho más claro. Si no sabes cómo probarlo, es porque está mal escrito.

---

# ⏳ Tipos de Requerimientos

No todo lo que pide el cliente dura para siempre:
* **Duraderos:** Son el ADN del negocio (ej. en una tienda, siempre habrá productos y ventas). Rara vez cambian.
* **Volátiles:** Los que cambian por modas, leyes nuevas o porque el cliente cambió de opinión a mitad del proyecto.

---

# 📂 ¿Dónde organizar la información?

Dependiendo del tamaño de tu equipo, puedes usar diferentes herramientas:
1.  **Básicas:** Excel o Word (baratas, pero se vuelven un caos si muchos meten mano).
2.  **Pro:** Herramientas como *IBM Rational*, *Jama* o *Visure*. Estas te dicen quién cambió qué, cuándo y por qué, manteniendo todo bajo control.

---

# 📈 Conclusión de Valor

La clave del éxito no está en programar rápido, sino en entender bien. Validar a tiempo es ahorrar dinero, evitar trasnochadas innecesarias y, sobre todo, entregar algo que el cliente realmente use y valore.
